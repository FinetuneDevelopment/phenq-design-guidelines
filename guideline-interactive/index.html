Tabs

Styling

Tab navigation is styled using the tab-set class. This has no impact on functionality.

Functionality

Documentation on how these tabs work can be found in this article.

The functional CSS relies on specific markup patterns:

  • The checkboxes must have an ID of tab1 up to tab10
  • The tab panels must have a role="tabpanel" attribute
  • IDs such as click-tabx should be replicated as-is
  • The label tags should have a for attribute which connects them to the correct radio button
  • The dynamic aria markup will be handled by tabs.js

Limitations

Unfortunately, the CSS only supports one tabset per-page. Multiple tab style navigation elements can be added, however, as can be seen above.

Mobile

If there is not enough room for the tabs to be displayed in one line, arrows will appear, allowing the user to scroll back and forward through the tab items.

The tabs will practice social distancing if there is enough room.

Scroller

Scrollers will take the width of their container and by default, their children will be 370px wide.

Child elements can take a width of 100% if a block-scroll__child-wide class is used directly on them. This means that each child will fit exactly inside the scrolling container and will be replaced with a different child, when the user interacts with the left and right buttons.

Accordions

Accordions come in two types:

Only one pane open at a time

Most of the functional aspects of the accordions are handled by CSS. However, there are numerous modifying classes used, which means the same functionality can be applied to accordions which look different.

For example, the only restriction placed by the CSS functionality is that the label (the accordion header) and the .details element follow the input element in the markup as siblings.

The JavaScript does three things:

  1. Updates the aria-expanded attribute, as the user interacts with the controls
  2. Allows the user to click on an open accordion header, in order to close it (as the CSS functionality is based on radio buttons, this requires the current radio button to be deselected, which is not possible with HTML)
  3. Attempts to scroll the opened accordion to the top of the viewport

If all of the hidden input elements are left unchecked, all panes of the accordion will be closed.

Add a checked attribute to the pane which should be open, when the user first lands on the page.

If more than one input element has a checked attribute, then the last element of that type will be open.

More than one pane can be open at a time

Change the type attribute of the input element from radio to checkbox.

Multiple accordion panes can be set to open by adding a checked attribute to any of these checkboxes.

Peekaboo

Description

Peekaboo panels allow the addition of large amounts of content into the page, most of which is hidden away.

The user is able to see a few lines of the content, before it fades out into the background colour.

Users can toggle this extra content on and off with a button at the bottom of the panel.

The fade at the bottom of the panel should match that of any solid background colour in use on the site.

Markup

An input type checkbox controls the toggling of the panel (the functionality would presumably work with radio, but this is untested. The input should appear first in the markup.

A following sibling with the class peekaboo-target contains the content to be hidden. This will have a max-height, so there will need to be sufficient content for the excess to be hidden.

A second following sibling with the class peekaboo-switch-dad contains a label tag which triggers the change in state in the checkbox. This label tag can be styled in any required way but note that the CSS will add text using before and update that text with the current state of the peekaboo panel.

The ID used to trigger the checkbox can be any valid string - this does not appear in the CSS.
